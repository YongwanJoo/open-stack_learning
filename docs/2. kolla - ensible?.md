# 2. Kolla-Ansible 및 OpenStack 핵심 개념 정리

이 문서는 OpenStack을 처음 접하는 사람도 쉽게 이해할 수 있도록, 각 기술이 무엇이고 어떤 역할을 하는지 쉽게 정리한 문서입니다.

---

## 1. OpenStack 핵심: Nova, Neutron, Keystone
OpenStack은 수많은 서버(컴퓨터)들을 모아서 거대한 '클라우드'로 만들어주는 **운영 체제**입니다. 

* **Nova**
  * **역할**: 사용자가 원할 때마다 가상 머신(VM)을 만들어주는 핵심 엔진
  * **비유**: AWS의 EC2와 같음
  
* **Neutron**
  * **역할**: 만들어진 가상 머신들에 IP 주소를 할당하고, 인터넷 선을 연결해주는 네트워크 서비스
  * **SDN (Software Defined Networking)**: 물리적인 랜선이나 라우터 장비를 직접 건드리지 않고, 소프트웨어 코드만으로 가상 네트워크를 마음대로 찍어내고 관리하는 기술
  * **비유**: AWS의 VPC와 같음

* **Keystone**
  * **역할**: 출입 및 권한 통제소
  * **RBAC (Role-Based Access Control)**: 사장님, 관리자, 일반 직원 등 각자의 '역할'에 따라 사용할 수 있는 권한과 접근할 수 있는 영역을 다르게 제어하는 보안 방식
  * **비유**: AWS의 IAM과 같음

---

## 2. Kolla-Ansible이란?
OpenStack은 기능이 너무 많아서 사람이 일일이 설치하려면 며칠이 걸리고 오류가 많기 때문에 terraform과 같이 간편하게 설치 가능한 도구가 필요하고 이 도구를 kolla-ansible 이라고 함

* **Kolla**
  * **역할**: 지저분하고 복잡한 OpenStack 프로그램들을 '컨테이너'로 격리하여 설치하는 역할
  * **장점**: Openstack을 컨테이너로 실행해서 컴퓨터에 컨테이너만 딱 올려놓으면 프로그램이 실행됨

* **Ansible**
  * **역할**: 사람이 직접 서버에 들어가서 타자를 치는 대신, 스크립트만 써두면 알아서 수십 대의 서버에 들어가 대신 타자를 치고 설치해 주는 자동화 도구
  * **멱등성 (Idempotency)**: 똑같은 설치 스크립트를 10번, 100번 반복해서 실행하더라도 시스템이 망가지지 않고 항상 동일한 결과 상태를 유지하도록 보장하는 아주 똑똑한 성질
  * **비유**: Kolla가 설치한 컨테이너들을 실행시켜주는 자동화 도구

---

## 3. Docker vs Podman
컨테이너들을 실행시키려면 컨테이너 엔진이 필요함

* **Docker**
  * 가장 유명하고 대중적인 컨테이너 엔진

* **Podman**
  * Docker의 단점을 보완해서 나온 최신형 엔진
  * Docker는 무조건 "최고 관리자(Root) 권한"이 있어야 시동이 걸리는데, Podman은 일반 직원도 시동을 걸 수 있어서 **보안이 훨씬 뛰어남**
  * **Daemonless (데몬리스)**: Docker처럼 항상 뒤에서 켜져 있으면서 자원을 갉아먹고 뻗으면 다 같이 죽는 중앙 관리자(데몬) 없이, 필요할 때만 독립적으로 가볍게 실행되는 아키텍처
  * 최근 OpenStack 생태계에서 공식적으로 밀고 있는 최신 트렌드

> **결론**: Kolla-Ansible이 알아서 컨테이너를 관리해주기 때문에 사용자 입장에서는 둘 다 비슷하게 생겼음

## 4. Kolla-Ansible 구축 과정 요약

**Ubuntu 22.04** 환경에 **Podman**과 **Kolla-Ansible(2024.1 Caracal)**을 사용하여 OpenStack All-in-One 환경을 구축

### Step 1. 인프라 준비 및 두 번째 랜카드 부착
* OpenStack 가상 네트워크(Neutron)가 외부 인터넷망과 통신하기 위해서는 물리적으로 독립된 네트워크 인터페이스가 필요함
* Terraform 코드를 수정하여 OCI 가상 머신에 두 번째 VNIC(`ens4`)를 부착

### Step 2. 사전 필수 패키지 설치
* Kolla-Ansible 구동 및 통신 암호화를 위해 Python 개발 도구(`python3-dev`), 가상 환경 모듈(`python3-venv`), C 컴파일러(`gcc`), 암호화 라이브러리(`libssl-dev`) 등을 OS에 설치

### Step 3. Kolla-Ansible 설치
* OS 환경이 꼬이는 것을 방지하기 위해 Python 가상 환경(`kolla-venv`)을 생성
* 가상 환경 내부에 패키지 매니저(`pip`)를 통해 Kolla-Ansible 2024.1 버전을 설치 (현재 공식 지원이 종료되어 `unmaintained/2024.1` 브랜치 사용)

### Step 4. 핵심 환경 설정 (`globals.yml`)
Kolla-Ansible이 OpenStack을 어떻게 배포할지 결정하는 가장 중요한 설정 파일
* **비밀번호 생성**: `kolla-genpwd` 명령어로 서비스 간 통신에 필요한 인증 암호들을 일괄 자동 생성
* **컨테이너 엔진**: Docker 대신 **Podman** 채택 (`kolla_container_engine: "podman"`)
* **네트워크 매핑**:
  * `network_interface`: `ens3` (OpenStack 내부 관리 및 API 통신용)
  * `neutron_external_interface`: `ens4` (가상 머신들의 외부 인터넷 통신용 랜카드)
  * `kolla_internal_vip_address`: 관리망 인터페이스(`ens3`)의 내부 IP를 할당하여 통신 엔드포인트 구성

### Step 5. OpenStack 자동 배포 (Playbook 실행)
* `bootstrap-servers`: 서버에 필요한 패키지(Podman 등)를 자동으로 설치하고 환경 세팅
* `prechecks`: 포트 충돌이나 설정 오류가 없는지 배포 전 최종 검사
* `deploy`: 수십 개의 OpenStack 컨테이너 이미지(Nova, Neutron 등)를 다운받고 실제로 구동
* `post-deploy`: 클라우드 제어를 위한 최고 관리자 로그인 인증 파일(`admin-openrc.sh`) 자동 생성

### Step 6. SSH 터널링을 통한 Horizon 대시보드 접속
* 보안을 위해 OpenStack 대시보드는 공인 IP가 아닌 내부 IP(10.0.0.171)로만 열어두어 외부 접근을 차단
* 로컬 브라우저에서 `http://localhost:8080`을 통해 접속하여, 프라이빗 클라우드 시각화 환경(Horizon) 진입
