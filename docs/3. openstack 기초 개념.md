# 3. OpenStack 기초 개념 및 아키텍처 이해

이 문서는 OpenStack이 도대체 무엇이며, 왜 사용하는지, 그리고 전체적인 구조가 어떻게 생겼는지 파악하기 위해 작성된 기초적인 내용을 담았습니다.

---

## 1. OpenStack이란 무엇인가?
* **한 줄 요약**: "데이터 센터 내의 방대한 컴퓨팅, 스토리지, 네트워킹 자원을 하나의 거대한 풀(Pool)로 묶어서 관리하고, 이를 사용자에게 가상 머신(VM) 형태로 쪼개어 제공하는 **클라우드 운영 체제(Cloud Operating System)**"
* **IaaS의 표준**: AWS, GCP, Azure 같은 퍼블릭 클라우드 환경을 **내 소유의 서버(온프레미스)**에 직접 구축할 수 있게 해주는 오픈소스 IaaS(Infrastructure as a Service) 플랫폼

## 2. 왜 OpenStack을 사용하는가?
* **비용 절감**: 퍼블릭 클라우드를 대규모로 사용할 때 발생하는 월 사용료를 절감하기 위해 자체 프라이빗 클라우드를 구축함
* **데이터 보안 및 규제**: 금융권, 공공기관 등 데이터를 외부 클라우드에 둘 수 없는 엄격한 보안 규제를 따르면서도 클라우드의 유연성(빠른 자원 할당)을 누리기 위함
* **벤더 종속(Vendor Lock-in) 탈피**: 특정 클라우드 사업자(AWS, MS)의 기술에 묶이지 않고, 독립적이고 개방적인 자체 인프라 환경을 구축하기 위함

---

## 3. OpenStack 핵심 아키텍처 및 AWS와의 비교
OpenStack은 하나의 단일 프로그램이 아니라, 수십 개의 독립적인 '프로젝트(서비스)'들이 레고 블록처럼 모여서 동작하는 구조로 각 컴포넌트는 AWS의 서비스들과 1:1로 매칭해 보면 이해가 쉬움

| OpenStack 서비스명 | 역할 및 기능 요약 | AWS 매칭 서비스 |
| :--- | :--- | :--- |
| **Nova** | **컴퓨트 서비스**: 가상 머신(VM)을 생성, 삭제 및 CPU와 RAM 자원을 스케줄링 | **Amazon EC2** |
| **Neutron** | **네트워킹 서비스**: 가상 스위치, 라우터, 방화벽, 로드밸런서 등을 소프트웨어(SDN)로 구축하여 VM에 네트워크 연결 | **Amazon VPC** |
| **Keystone** | **인증 서비스**: 클라우드에 접속하는 사용자의 신원(ID/PW)을 확인하고, 어떤 서비스에 접근할 수 있는지 권한 통제 | **AWS IAM** |
| **Glance** | **이미지 서비스**: VM을 띄울 때 필요한 운영체제 이미지(Ubuntu, CentOS, Windows 등)를 보관 및 제공 | **Amazon AMI** |
| **Cinder** | **블록 스토리지**: VM에 추가로 탈부착할 수 있는 가상 하드디스크(Volume) 제공 | **Amazon EBS** |
| **Swift** | **오브젝트 스토리지**: 파일, 비디오, 백업 데이터 등 비정형 데이터를 URL을 통해 무제한으로 저장하고 가져오는 서비스 | **Amazon S3** |
| **Horizon** | **대시보드**: 위 모든 서비스들을 터미널 명령어(CLI)가 아닌, 웹 브라우저 화면에서 마우스 클릭으로 제어할 수 있게 해주는 UI | **AWS Management Console** |
| **Heat** | **오케스트레이션 (IaC)**: 인프라를 수동으로 만들지 않고, 템플릿(코드)을 작성하면 수백 대의 서버와 네트워크를 한 번에 자동 배포 | **AWS CloudFormation** |

## 4. 서비스 간의 협력 방식 (VM 생성 흐름 예시)
사용자가 Horizon(대시보드)에서 "우분투 서버 1대 만들어줘!"라고 클릭하면 내부적으로 다음과 같은 일이 일어납니다.
1. **Horizon**: 사용자의 클릭을 API 요청으로 변환하여 **Keystone**을 통해 권한 확인
2. **Keystone**: "인증 완료 -> 서버 생성 권한 확인 완료"
3. **Nova**: 서버 생성 요청을 받고 가상 머신 준비
4. **Glance**: Nova의 요청을 받아 보관 중이던 '우분투 이미지'를 Nova에게 전달
5. **Neutron**: Nova의 요청을 받아 생성된 가상 머신에 랜선을 꽂고 내부 IP 주소를 할당
6. **Cinder** (선택): 가상 머신에 100GB짜리 추가 하드디스크를 포맷해서 연결

이 모든 과정을 오픈스택 환경 내부의 **RabbitMQ**를 통해 비동기적으로 처리
